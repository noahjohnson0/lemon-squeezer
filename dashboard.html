🍋 The Lemon Squeezer

Get the absolute most juice possible out of a local LLM.

Reproducible benchmarks for local-LLM coding agents across heterogeneous hardware — currently an RTX 4070 (12 GB) Ollama host, with M4 Max (48 GB) and a clustered configuration on the runway. Same prompts and rubrics, different harnesses, configs, models, and machines. The premise: cloud benchmarks tell you almost nothing about whether qwen3-coder on hardware you actually own can finish a real-world task. This finds out.

Headline finding so far: same qwen3:14b on the bug-fix eval scores 27% through pi and 100% through aider. The harness matters more than the model — pi's exact-string-match edits fail for small models that hallucinate file contents.

Leaderboard –

Best score per (harness × model × eval). Each cell shows the highest score that combination has achieved across any tag (baseline / sysprompt / directive / etc.). The rank column is a Bayesian-shrunk score — (n·avg + C·μ) / (n + C), with C=3 pseudo-evals' worth of pull toward the global mean. A 1/8 row of 100% won't outrank a 7/8 row of 95%. Hardware telemetry –

Sampled every 1s during each run via nvidia-smi on the inference host. We capture GPU utilisation, VRAM used, temperature, power draw, fan speed, and host load average. Only runs after the sampler was added carry telemetry.

Per-model averages

Across all runs of a model — average GPU temp, peak VRAM, average power. Models that consistently push the GPU harder cost more electricity per token.

Latest run telemetry

Time series of GPU temperature, power draw, and utilisation across the most recent telemetered run.

Prompt-config effect tag impact

Same model on the same eval across config tags. baseline is no extra prompting; sysprompt adds configs/general-hygiene.md via aider --read; directive is more prescriptive; comprehensive bundles everything. Bigger models often get less benefit — they already know.

Per-eval deep dive –

Each eval is a self-contained directory with a prompt.md, an optional setup.sh that drops starter files, and a rubric.sh that runs the produced code and scores it. Click any row to see the per-check breakdown for that run.
